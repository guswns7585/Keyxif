# Keyxif 커스텀 템플릿 에디터 구현 계획

이 문서는 Keyxif 프로젝트에 커스텀 템플릿 생성 기능을 순차적으로 구현하기 위한 기준 문서다.

코덱스는 작업을 시작할 때 이 파일을 먼저 읽고, 현재 완료된 단계와 다음 작업 단계를 확인해야 한다.
각 단계가 완료될 때마다 이 문서 하단의 진행 상태와 작업 기록을 갱신한다.

---

## 0. 핵심 목표

기존 템플릿 선택 방식에 더해, 사용자가 직접 사진과 정보 요소의 레이아웃을 구성하고 저장할 수 있는 커스텀 템플릿 에디터를 추가한다.

커스텀 템플릿 하나에서 다음 기능을 동시에 사용할 수 있어야 한다.

1. 사진 뒤에 배경 프레임을 생성하고 크기와 비율을 자유롭게 조절
2. 배경 프레임 안에서 사진의 위치와 크기를 조절
3. 사진 외부 공간에 텍스트, 로고, 색상칩을 자유롭게 배치
4. 사진 내부에 텍스트, 로고, 색상칩을 자유롭게 배치
5. 사진 내부에 여러 개의 커스텀 카드를 생성하고 배치
6. 내부 카드 안에 텍스트, 로고, 색상칩을 자유롭게 배치
7. 완성된 커스텀 템플릿을 저장하고 기존 템플릿처럼 선택·적용
8. 기존 색상 단계에서 프레임 및 카드 배경색 등 연결된 색상을 변경

---

## 1. 절대 조건

1. 원본 사진은 절대 크롭하지 않는다.
2. 사진의 종횡비는 항상 유지한다.
3. 사진은 contain 방식으로 전체가 표시되어야 한다.
4. 커스텀 템플릿 기능 때문에 기존 기본 템플릿과 기존 변환 기능이 깨지면 안 된다.
5. 좌표와 크기는 출력 해상도와 무관하게 재현될 수 있도록 정규화 좌표를 사용한다.
6. 미리보기와 최종 출력은 동일한 공통 레이아웃 계산 로직을 사용한다.
7. 요소는 자신이 소속된 컨테이너를 명확하게 가져야 한다.
8. 사진 외부 요소는 사진과 겹치거나 경계에 걸칠 수 없다.
9. 내부 카드 요소는 카드 밖으로 렌더링되면 안 된다.
10. 구현 중 발견되는 버그는 임시 우회보다 원인 분석과 구조적 수정을 우선한다.
11. 각 단계는 독립적으로 빌드와 기본 동작 검증이 가능해야 한다.
12. 한 번에 전체 기능을 구현하지 않는다.

---

## 2. 용어 및 레이어 구조

### 2.1 UI 탭 이름

- 프레임
- 요소
- 카드

### 2.2 의미

프레임:
- 출력 캔버스
- 사진 뒤 배경 프레임
- 프레임 비율과 크기
- 사진 위치와 크기
- 프레임 배경색

요소:
- 텍스트
- 로고
- 색상칩

카드:
- 사진 내부에 배치되는 커스텀 카드
- 카드 스타일
- 카드 내부 콘텐츠

### 2.3 권장 레이어 순서

1. 배경 프레임
2. 원본 사진
3. 배경 프레임 소속 요소
4. 사진 내부 카드
5. 사진 소속 자유 요소
6. 선택선, 핸들, 스냅 가이드 등 편집 전용 UI

편집 전용 UI는 최종 이미지에 포함하면 안 된다.

---

## 3. 배경 프레임 구조

상·하·좌·우 외부 영역을 각각 생성하는 방식은 사용하지 않는다.

사진보다 뒤에 하나의 사각형 배경 프레임을 배치한다.

사용자는 다음을 조절할 수 있다.

- 프레임 너비
- 프레임 높이
- 프레임 종횡비
- 프레임 배경색
- 프레임 안에서 사진 위치
- 프레임 안에서 사진 크기

사진 외부 영역은 다음과 같이 계산한다.

외부 콘텐츠 영역 = 배경 프레임 영역 - 사진 영역 - 사진 주변 안전 여백

이 방식으로 사진의 상하좌우뿐 아니라 네 모서리 대각선 공간도 하나의 연속된 외부 영역으로 사용한다.

사진은 프레임 안에서 좌상단, 중앙, 우하단 등 원하는 위치로 이동할 수 있어야 한다.

프레임은 사진 전체를 포함할 수 있는 최소 크기보다 작아질 수 없다.

---

## 4. 정보 필드

커스텀 템플릿의 텍스트 요소는 다음 데이터 필드에 연결할 수 있다.

- board
- switch
- plate
- mount
- nickname

로고 요소는 다음 데이터에 연결한다.

- logo image

색상칩은 사용자가 지정한 색상 또는 현재 사진에서 추출된 색상 데이터와 연결할 수 있도록 확장 가능한 구조로 설계한다.

텍스트 표시 형식은 다음 형태를 지원할 수 있어야 한다.

- 값만 표시: NEO65
- 레이블과 값: BOARD · NEO65
- 콜론 형식: Board: NEO65

초기 구현에서는 값만 표시를 기본값으로 사용해도 된다.

---

## 5. 권장 데이터 모델

아래 구조를 현재 프로젝트의 언어와 상태 관리 방식에 맞게 조정하되, 개념은 유지한다.

```ts
type CustomTemplate = {
  id: string;
  templateVersion: number;
  name: string;
  createdAt: string;
  updatedAt: string;

  frame: BackgroundFrame;
  photoPlacement: PhotoPlacement;

  elements: CanvasElement[];
  internalCards: InternalCard[];

  thumbnail?: string;
};

type BackgroundFrame = {
  logicalWidth: number;
  logicalHeight: number;
  aspectRatio: number;

  fill: {
    type: "solid";
    color: string;
    colorSlotId?: string;
  };
};

type PhotoPlacement = {
  x: number;
  y: number;
  width: number;
  height: number;
  scale: number;
  aspectRatio: number;
  fitMode: "contain";
  safePadding: number;
};

type CanvasElement = {
  id: string;
  type: "text" | "logo" | "colorChip";

  containerId: string;
  coordinateSpace: "frame" | "photo" | "internalCard";

  x: number;
  y: number;
  width: number;
  height: number;

  rotation: number;
  zIndex: number;

  locked: boolean;
  hidden: boolean;

  style: ElementStyle;
  content: ElementContent;
};

type InternalCard = {
  id: string;
  containerId: "photo";

  x: number;
  y: number;
  width: number;
  height: number;

  zIndex: number;
  locked: boolean;
  hidden: boolean;

  style: CardStyle;
};

type CardStyle = {
  backgroundColor: string;
  colorSlotId?: string;
  opacity: number;
  radius: number;

  borderEnabled: boolean;
  borderColor: string;
  borderWidth: number;

  padding: number;

  shadowEnabled: boolean;
  shadowBlur: number;
  shadowOpacity: number;
};
```

중요:
- 프레임은 논리 해상도와 종횡비를 가진다.
- 프레임 내부 요소와 사진 배치는 0~1 정규화 좌표를 사용한다.
- 사진 내부 요소는 사진 기준 정규화 좌표를 사용한다.
- 카드 내부 요소는 해당 카드 기준 정규화 좌표를 사용한다.
- `templateVersion`은 반드시 저장한다.
- 기존 템플릿을 위한 마이그레이션 또는 기본값 보정 함수를 준비한다.

---

## 6. 공통 에디터 규칙

모든 배치 가능한 요소는 가능한 한 동일한 편집 시스템을 사용한다.

공통 기능:

- 선택
- 선택 해제
- 드래그 이동
- 모서리 핸들 크기 조절
- 복제
- 삭제
- 앞쪽으로 이동
- 뒤쪽으로 이동
- 잠금
- 정렬
- 방향키 미세 이동
- Undo
- Redo

초기 MVP에서 제외 가능:

- 자유 회전 UI
- 다중 선택
- 그룹화
- 복잡한 자동 레이아웃
- 실시간 컨테이너 간 자유 이동

회전 필드는 데이터 모델에는 유지할 수 있다.

Undo/Redo는 포인터 이동 매 프레임을 저장하지 않는다.
조작 시작 전 상태와 조작 종료 후 상태만 기록한다.

입력 필드에 포커스가 있을 때 Delete 또는 Backspace로 요소가 삭제되면 안 된다.

# 단계별 구현 계획

---

## 단계 0. 프로젝트 분석

### 목표

코드를 수정하기 전에 현재 프로젝트 구조와 기존 템플릿 처리 방식을 정확하게 파악한다.

### 작업

1. 템플릿 선택 화면 위치 확인
2. 템플릿 데이터 모델 확인
3. 사진 미리보기 렌더링 방식 확인
4. 최종 이미지 출력 방식 확인
5. 색상 단계와 템플릿 연결 방식 확인
6. 로고 및 빌드 정보 데이터 흐름 확인
7. 로컬 저장 또는 프로젝트 저장 방식 확인
8. 관련 파일과 컴포넌트 목록 작성
9. 기존 구조에서 발생할 수 있는 충돌과 위험 요소 분석
10. 구현 계획을 현재 코드 구조에 맞게 조정

### 완료 조건

- 코드를 수정하지 않은 분석 보고서가 작성되어야 한다.
- 수정 예상 파일 목록이 있어야 한다.
- 데이터 흐름과 렌더링 흐름이 설명되어야 한다.
- 단계 1 구현 전 확인이 필요한 항목이 없어야 한다.

---

## 단계 1. 데이터 모델과 편집 화면 뼈대

### 목표

커스텀 템플릿 데이터를 정의하고 편집 화면의 기본 구조를 만든다.

### 작업

1. CustomTemplate 데이터 모델 추가
2. 버전 필드 추가
3. 임시 편집 상태 추가
4. 템플릿 선택 화면 상단에 `템플릿 만들기` 버튼 추가
5. 버튼 클릭 시 편집 화면으로 전환
6. 편집 화면 상단에 미리보기 영역 배치
7. 편집 화면 하단에 `프레임 / 요소 / 카드` 탭 배치
8. 탭에 단순한 솔리드 아이콘 적용
9. 빈 템플릿 초기값 생성
10. 화면 이탈 및 뒤로가기 기본 흐름 구성

### 이번 단계에서 제외

- 실제 드래그
- 요소 생성
- 프레임 리사이즈
- 저장
- 최종 출력

### 완료 조건

- 템플릿 만들기 버튼으로 편집 화면 진입 가능
- 빈 미리보기 표시
- 세 개 탭 전환 가능
- 타입 검사 및 빌드 성공
- 기존 템플릿 선택 화면 정상 동작

---

## 단계 2. 공통 캔버스 조작 기반

### 목표

향후 프레임, 카드, 텍스트, 로고, 색상칩이 공통으로 사용할 편집 기반을 만든다.

### 작업

1. 논리 좌표와 화면 좌표 변환 함수 작성
2. 0~1 정규화 좌표 적용
3. 테스트용 사각형 요소 렌더링
4. 요소 선택과 선택 해제
5. 드래그 이동
6. 모서리 핸들 크기 조절
7. 최소 크기 제한
8. 허용 컨테이너 범위 제한
9. z-index 렌더링
10. 삭제
11. Undo/Redo
12. 마우스와 터치 포인터 이벤트 처리
13. 브라우저 기본 드래그 및 텍스트 선택 방지
14. 리사이즈 후 상대 위치 유지

### 완료 조건

- 테스트 요소 선택 가능
- 이동 및 크기 조절 가능
- 경계 밖으로 완전히 이탈하지 않음
- Undo/Redo 정상 작동
- 화면 크기 변경 후 위치 유지
- 기존 화면 회귀 문제 없음

---

## 단계 3. 배경 프레임과 사진 배치

### 목표

사진 뒤 배경 프레임을 만들고 사진을 크롭 없이 배치한다.

### 핵심 원칙

상하좌우 외부 영역을 각각 만들지 않는다.

하나의 배경 프레임과 그 위에 배치되는 사진 구조를 사용한다.

### 작업

1. 배경 프레임 생성
2. 프레임 너비와 높이 조절
3. 프레임 종횡비 조절
4. 모서리 및 변 리사이즈 핸들
5. 자주 사용하는 종횡비 프리셋을 추가할 수 있는 구조 준비
6. 사진을 프레임 안에 contain 방식으로 표시
7. 사진 원본 종횡비 유지
8. 프레임 안에서 사진 이동
9. 사진 크기 조절
10. 사진이 프레임 밖으로 나가지 않도록 제한
11. 프레임이 사진보다 작아지지 않도록 제한
12. 사진 외부 영역 계산
13. 사진 주변 safe padding 계산
14. 편집 중 외부 사용 가능 영역 오버레이
15. 프레임 임시 배경색 설정
16. 향후 색상 단계 연결용 color slot 구조 추가

### 외부 영역 계산

- 외부 콘텐츠 영역은 `프레임 - 사진 - safe padding`이다.
- 사진의 대각선 바깥 공간도 외부 영역에 포함한다.
- 별도의 top, bottom, left, right 영역을 만들지 않는다.

### 완료 조건

다음 시나리오가 정상이어야 한다.

1. 사진 중앙 배치
2. 사진 좌측 배치
3. 사진 상단 배치
4. 사진 좌상단 배치
5. 가로 사진
6. 세로 사진
7. 정사각형 사진
8. 프레임 종횡비 변경
9. 사진 크롭 미발생
10. 프레임과 사진 위치 저장 데이터 정상

---

## 단계 4. 텍스트, 로고, 색상칩

### 목표

사용자가 요소 탭에서 콘텐츠를 추가하고 프레임 또는 사진 내부에 자유롭게 배치할 수 있게 한다.

### 추가 방식

1. 항목을 캔버스로 드래그해 추가
2. 항목을 클릭해 선택된 컨테이너 중앙에 추가

모바일과 터치 환경 때문에 클릭 추가를 반드시 함께 제공한다.

### 텍스트

지원 데이터:

- board
- switch
- plate
- mount
- nickname

지원 속성:

- 고정 문자열 또는 데이터 필드 연결
- 폰트 크기
- 글자 굵기
- 글자색
- 정렬
- 줄 간격
- 글자 간격
- 최대 줄 수
- 자동 줄바꿈
- 레이블 표시 방식
- 필요 시 대문자 변환

### 로고

- logo image 연결
- 투명 PNG 지원
- 원본 비율 유지
- contain 방식
- 왜곡 금지

### 색상칩

- 원형
- 사각형
- 둥근 사각형
- 컬러피커
- 크기
- radius
- 복제

### 컨테이너

드롭 위치에 따라 다음 중 하나에 소속한다.

- frame
- photo
- internalCard

단, internalCard는 단계 5에서 구현한다.

### 사진 경계 규칙

프레임 요소는 사진 위에 겹칠 수 없다.

- 요소 bounding box가 사진 safe padding 영역을 침범하면 안 된다.
- 경계 가까이 이동하면 허용 영역으로 스냅한다.
- 경계에 걸친 상태로 저장하면 안 된다.
- 사진 내부 요소는 사진 좌표 공간에 소속한다.
- 프레임 외부 요소는 프레임 좌표 공간에 소속한다.

### 완료 조건

- 텍스트 추가 및 데이터 연결
- 로고 추가
- 색상칩 추가
- 이동 및 크기 조절
- 복제와 삭제
- 프레임과 사진 좌표 공간 구분
- 프레임 요소의 사진 침범 방지
- 투명 로고 정상 표시

## 단계 5. 사진 내부 커스텀 카드

### 목표

사진 내부에 여러 개의 카드를 생성하고 각 카드 안에 요소를 배치할 수 있게 한다.

### 카드 스타일

- 배경색
- 투명도
- radius
- 테두리 사용 여부
- 테두리 색
- 테두리 두께
- 내부 padding
- 그림자 사용 여부
- 그림자 blur
- 그림자 투명도
- 향후 backdrop blur 확장 가능 구조

### 카드 생성 방식

1. 카드 탭에서 카드 스타일 미리보기 제공
2. 카드 미리보기를 사진으로 드래그해 생성
3. 클릭 추가 방식 제공
4. 사진 내부에서만 배치
5. 크기와 비율 자유 조절
6. 최소 크기 제한
7. 카드끼리 겹침 허용
8. z-index 조절
9. 개수는 임의의 낮은 제한을 두지 않음

### 카드 내부 요소

- 텍스트
- 로고
- 색상칩
- 카드 기준 정규화 좌표 사용
- 카드 padding을 제외한 콘텐츠 영역 안에 배치
- 카드 밖 렌더링 금지
- overflow hidden 적용
- 카드 크기 변경 시 내부 요소 상대 위치 유지

### 삭제

- 카드 삭제 시 내부 요소도 함께 삭제
- 삭제 전 확인 또는 Undo 제공
- Undo 시 카드와 내부 요소 모두 복원

### 완료 조건

- 여러 카드 생성
- 카드 이동 및 크기 조절
- 카드 겹침
- 카드 스타일 변경
- 카드 내부 요소 배치
- 카드 삭제와 Undo
- 카드 밖 콘텐츠 렌더링 방지

## 단계 6. 스냅과 경계 충돌 처리

### 목표

사진과 프레임 경계, 카드 경계, 다른 요소와의 정렬을 안정적으로 처리한다.

### safe padding과 gap 구분

- gap: 사진과 외부 배경 사이에 실제로 보이는 간격
- safePadding: 요소가 경계에 너무 가까워지지 않도록 하는 편집 제한

두 값은 데이터 구조에서 분리한다.

### 스냅 우선순위

1. 컨테이너 안전 영역
2. 컨테이너 중앙선
3. 컨테이너 모서리
4. 다른 요소의 좌우 정렬선
5. 다른 요소의 상하 정렬선

### 동작

- 임계값 안에서만 스냅
- 스냅 가이드 표시
- 강제로 느껴지지 않게 설정
- 보조 키 또는 향후 설정으로 스냅 일시 해제 가능
- 사진 위치 변경으로 기존 프레임 요소와 충돌 시 경고
- 충돌 요소를 임의 삭제하지 않음
- 예측 불가능한 위치로 강제 이동하지 않음
- 저장 전 사용자가 충돌을 해소하도록 요구

### 완료 조건

- 사진 경계 스냅
- 카드 내부 경계 제한
- 중앙선 스냅
- 요소 간 정렬 스냅
- 충돌 경고
- 경계에 걸친 상태 저장 방지

## 단계 7. 카드 내부 공간 부족 검사

### 목표

카드 내부 정보가 카드 영역을 초과하거나 읽기 어려운 상태를 감지한다.

### 검사 항목

1. 텍스트 실제 렌더링 영역 초과
2. 로고 경계 초과
3. 색상칩 경계 초과
4. 카드 padding 침범
5. 최대 줄 수 초과
6. 최소 글자 크기 미만
7. 치명적인 요소 겹침
8. 요소가 카드 콘텐츠 영역 밖에 존재

### 경고 방식

- 최초 발생 시 토스트:
  `공간이 부족합니다. 카드 크기를 늘리거나 요소를 줄여주세요.`
- 같은 문제로 토스트 반복 금지
- debounce 또는 상태 기반 중복 방지
- 문제가 있는 카드에 경고 테두리 또는 아이콘 표시
- 저장 시 전체 재검사
- 치명적 경계 초과 시 저장 차단
- 문제가 있는 카드를 자동 선택
- 단순 겹침은 경고만 하고 저장 허용 가능

### 텍스트 자동 축소

- 기본 비활성화
- 사용자가 명시적으로 켰을 때만 제한 범위 내 허용
- 최소 폰트 크기 설정
- 최소 크기보다 더 작아져야 들어가면 공간 부족으로 판정

### 구현 원칙

검증 로직은 UI 컴포넌트에 흩어놓지 않는다.
순수 함수와 공통 유틸리티로 분리한다.

### 완료 조건

- 작은 카드와 긴 텍스트
- 큰 로고
- 여러 요소
- padding 침범
- 토스트 중복 방지
- 저장 차단 및 문제 카드 선택
- 카드 밖 렌더링 방지

- 카드 배경색

- 색상칩

고정 색상으로 설정된 요소는 저장된 색상을 유지한다.

- 키보드 이벤트가 입력 필드에 영향을 주지 않도록 Web에서는 input/select/textarea에서 단축키를 무시하고, Android에서는 편집 캔버스 포커스 영역에만 키 이벤트를 연결
- 선택 대상 이동/정렬/초기화는 기존 normalized 좌표, snap, card space validation, undo/redo 구조를 재사용하도록 구현
- 기존 기본 `CardTemplate` 렌더러, 기본 템플릿 선택, 이미지 저장/변환 경로는 변경하지 않음
- 검증 완료: Android `:app:compileDebugKotlin`, `:app:testDebugUnitTest`, `:app:assembleDebug`; Web `node --check ui-template-editor.js`, `node --check store.js`, `node --check ui-main.js`, `npm test`, `npm run build`; `git diff --check`는 CRLF 변환 경고만 표시
- 커스텀 템플릿 에디터 계획의 단계 0~10 구현을 모두 완료함

# 금지 사항

1. 전체 기능을 한 번에 구현하지 않는다.
2. 현재 단계보다 앞선 기능을 임의로 추가하지 않는다.
3. 기존 기본 템플릿 구조를 불필요하게 전면 교체하지 않는다.
4. 사진을 cover 방식으로 렌더링하지 않는다.
5. 사진을 크롭하지 않는다.
6. 픽셀 절대 좌표만 저장하지 않는다.
7. 프레임 외부 영역을 상하좌우 네 개의 독립 컨테이너로 분리하지 않는다.
8. 프레임 요소가 사진 경계에 걸친 상태로 저장되게 하지 않는다.
9. 카드 내부 요소가 카드 밖으로 보이게 하지 않는다.
10. 공간 부족 경고 토스트를 매 프레임 반복 표시하지 않는다.
11. 드래그 이벤트마다 Undo 기록을 생성하지 않는다.
12. 테스트 없이 다음 단계로 넘어가지 않는다.
13. 기존 기능의 회귀 문제를 확인하지 않은 채 완료 처리하지 않는다.
14. 사용자의 명시적인 요청 없이 다음 단계를 연속 실행하지 않는다.
